How to use this document

This covers the entire navigation stack, end to end: from a single robot walking through static rubble, all the way to multiple robots sharing information with each other. It's organized as four phases, and each phase has the same four sections — why this phase exists, what architecture gets added, a day-by-day plan, and how the project directory changes. The goal isn't to hand you code to paste — it's to let you understand why each piece exists before you write it, so when something breaks (and something will break) you know what it's supposed to be doing.

A note on realistic timing

This is the full, research-grade version — the one with real vision perception, the proprioception feedback loop, multi-objective Pareto sweeps, and multi-robot communication. Realistically this is a 4 to 6 week body of work, not a few days. Each phase below states its own time estimate; treat the day-by-day breakdowns as "what a day of focused work on this phase looks like," not as a guarantee you'll finish a whole phase in the days shown if you're also doing coursework, debugging unrelated things, or this is your first time touching some of these tools.

How the stack grows, at a glance

Each phase adds exactly one new capability on top of the last. Nothing already working gets rebuilt — this is deliberate, so a failure in phase 3 doesn't put phase 1's progress at risk.

PHASE 01
Perceive + plan + act

Depth camera, A* over known map, PPO policy, frozen locomotion. One robot, still obstacles.

PHASE 02
+ react to motion

Obstacles move. Frame-stacking gives implicit velocity. A risk-prediction head anticipates collisions.

PHASE 03
+ trade off objectives

One reward becomes four (progress, energy, jerk, time). Train several weightings, compare the tradeoffs.

PHASE 04
+ talk to each other

Multiple robots, each running phases 1-3, now exchange compressed messages about what they see and intend.

Phase 01 · ~1.5-2 weeks

Static rubble navigation

A single Go2 walks from a start point to a goal through a field of static rubble — fallen pillars, debris, uneven terrain — using real depth-camera perception, not privileged sim state. This is the foundation every later phase sits on, so it's worth getting genuinely right rather than rushing.

Why build it this way

Why a frozen locomotion policy underneath, instead of training everything together? Your locomotion policy already knows how to walk — that problem is solved. If you let navigation training also update locomotion weights, every change to the navigation reward risks un-learning walking, and debugging becomes "is my new reward bad, or did I just break gait stability?" with no way to tell which. Freezing locomotion means navigation failures are unambiguously navigation failures.
Why A* for the global route instead of letting RL find the way? The rubble layout is something you build — it's known, in full, before the robot ever takes a step. RL is the right tool for things that are hard to specify in advance (how to react to a camera frame you've never seen before); it's the wrong, slow tool for "what's the shortest path through a maze I already have the blueprint for." A* solves that instantly. Save the learning budget for the part that actually needs it.
Why depth camera instead of just trusting the height-scan you already have? Your height-scan tells you about the ground under and near the robot's feet — it's a locomotion-scale sensor. A fallen pillar at head height, or a wall two meters ahead, isn't something a foot-level height-scan sees coming. The navigation policy needs to look outward, not just downward.

Architecture for Phase 01

sensing / mapping planning & control

Notice this is a straight pipeline, no loops yet. That's deliberate — phase 01's job is to prove every individual stage works in isolation before phase 02 adds the complexity of things moving, and phase 03 adds competing objectives. A straight pipeline is also the easiest to debug: if something's wrong, you can check each stage's output one at a time.

Day-by-day — Phase 01

Days 1-2

Scene + sensor, before any RL

Why start here: if the sensor data going into the policy is wrong, no amount of reward-tuning will fix it. Validate this in isolation, visually, before writing a single line of training code.
  • Build the rubble scene: terrain generator or hand-placed static obstacle primitives in navigation_env_cfg.py
  • Add a depth camera sensor definition to the scene config — check Isaac Lab's camera sensor API for resolution/FOV/range
  • Write the projection from depth image → egocentric occupancy grid as a standalone function first, test it on a single saved depth frame before wiring into the env
  • Checkpoint: visualize the occupancy grid next to a screenshot of the scene — do the occupied cells actually line up with where the obstacles are?
Days 3-4

A* planner, standalone

Why standalone: A* is pure graph search — it has nothing to do with Isaac Lab, physics, or RL. Writing and testing it as plain Python first means you can verify correctness with simple print statements and a matplotlib plot, instead of debugging it inside a slow simulation loop.
  • Implement grid A* over a coarse static-layout cost grid in planner.py — no Isaac Lab imports
  • Unit test: known small grid with a known obstacle, verify the returned path avoids it and is reasonably short
  • Only once that's solid, wire it into a new command term in commands.py that runs the planner on reset and exposes "current waypoint" each step
  • Checkpoint: in the Isaac Lab GUI, does a printed/visualized waypoint sequence trace a sensible route around the rubble?
Days 5-7

Wire the frozen locomotion policy

Why this is its own step, not bundled with reward-writing: this is the exact class of bug you already hit once (the play.py shape mismatch). Isolate it: prove you can load the frozen checkpoint and successfully call it with a hand-typed velocity command before any learned policy is in the loop at all.
  • Load the frozen locomotion checkpoint inside the new navigation env
  • Manually feed it a constant velocity command (e.g. "always walk forward") and confirm the robot walks correctly in the GUI — this isolates "is locomotion wired right" from "is navigation working"
  • Replace the constant command with a simple hand-coded rule (e.g. "always move toward the current waypoint") — still no RL — and confirm the robot visibly heads toward goals
  • Checkpoint: a non-learned, rule-based "policy" gets the robot most of the way to the goal in a simple scene. This proves the whole pipeline plumbing works before RL enters the picture.
Days 8-10

Observation + reward + first training runs

Why test reward scale before a long run: energy, progress, and collision penalties live on very different numeric scales. If one term is 100x larger than another, PPO will optimize almost entirely for that one term and ignore the rest — this looks like "the policy isn't learning" but is actually "the policy is learning the wrong thing perfectly."
  • Build the navigation_policy observation group: cost-map (small CNN encoder) + goal vector + robot base state, in observations.py
  • Implement reward terms in rewards.py: progress, heading alignment, collision penalty, smoothness — log each as a separate scalar from the start
  • Run a few hundred steps with random actions first, just to confirm reward magnitudes are sane and nothing crashes
  • Write the skrl PPO agent config for navigation (smaller network than locomotion — this is a lower-dimensional problem)
  • Launch a short (~30-60 min) training run and check TensorBoard — is anything trending in the right direction?
  • Checkpoint: reward is increasing, even slowly, and isn't NaN/exploding. Only then commit to a long run.
Days 11-14

Full training + validation

  • Run training to convergence (this is mostly waiting + periodic TensorBoard checks, plan other work alongside it)
  • Validate with play.py: does the robot reliably reach goals across several different start/goal pairs in the rubble scene?
  • Write eval_metrics.py: N-episode rollout logging time-to-goal, success rate, collision rate
  • Checkpoint, end of phase 01: a trained nav policy that gets a single robot through static rubble most of the time, with logged metrics as a baseline for later phases to compare against.

mdp/observations.py

Turns the depth camera's raw output into the egocentric occupancy grid, and computes the goal distance/bearing each step. This is the file most worth getting right — every later phase's perception additions build on top of what this returns.

mdp/commands.py

Replaces the locomotion env's random velocity-command sampler. Calls planner.py once per episode/goal, tracks which waypoint is "current," and on each step accepts the nav policy's action as the actual velocity passed to the frozen locomotion policy.

planner.py

Pure-Python A*, deliberately isolated from Isaac Lab so it can be tested on its own. If navigation ever behaves strangely, checking this file in isolation (does it return sane waypoints for a given map?) rules out a whole category of bugs fast.

eval_metrics.py

Loads a trained nav checkpoint plus the frozen locomotion checkpoint, runs N episodes, writes one row per episode. This script's output is your evidence that phase 01 actually works, and the same script gets extended (not replaced) in phases 02-03.

Phase 02 · ~1-1.5 weeks

Dynamic obstacles

People and objects now move through the scene — scripted motion is enough, they don't need their own intelligence. The navigation policy has to react to things it can't predict in advance, which is precisely the part A* can't help with, since the global route only knows about the static layout.

Why build it this way

Why frame-stacking instead of an explicit tracker? You could build a module that detects an obstacle, tracks its position over time, and estimates its velocity — that's a real, separate engineering project. Stacking the last 2-3 occupancy grids as channels gets you almost the same information for free: a static obstacle looks identical across all stacked frames, a moving one visibly shifts position between them. The CNN encoder you already built in phase 01 can learn to read that shift on its own, the same way frame-stacked Atari agents learn to perceive ball velocity without anyone building a tracker.
Why no replanning when an obstacle moves? A*'s output only depends on the static layout — rubble doesn't get up and walk away. The global route stays valid no matter what the dynamic obstacles do; only the local, reactive part of the system needs to respond to them. This is also why the system stays fast: there's no replanning loop running at all, just one forward pass through the nav policy per control step.
Why add a risk-prediction head instead of just a bigger collision penalty? A penalty that only fires after a collision teaches the policy "collisions are bad" but gives it no signal about when one is about to happen. A small auxiliary output trained to predict "will I collide in the next N steps" gives an anticipatory signal — the policy can learn to slow down or deflect before contact, not just be punished after. This is a scoped-down version of what Agile But Safe (ABS) does with a full reach-avoid value network; you're getting the same anticipate-and-back-off behavior from a much simpler auxiliary head.

What's added in Phase 02

sensing / mapping planning & control

Everything from phase 01 (A* planner, the base reward terms, the frozen locomotion policy) stays exactly as it was. The dashed box is new infrastructure that feeds the existing occupancy pipeline; the risk head is a new auxiliary output bolted onto the existing nav policy, not a replacement for it.

Day-by-day — Phase 02

Days 1-2

Scripted dynamic obstacles

Why scripted, not RL-controlled: giving the obstacles their own learned behavior turns this into a much harder multi-agent problem you don't need yet. Simple, predictable scripted motion is enough to force the nav policy to actually react to change, without you having to train a second agent just to get there.
  • Add 2-4 movable assets to the scene (boxes, simple humanoid placeholders) in the scene config
  • Write the motion script in events.py — bounded random walk or back-and-forth linear motion, reset on episode start
  • Checkpoint: in the GUI, do the obstacles visibly move in a sane, bounded way without flying off or clipping through walls?
Days 3-4

Frame stacking

  • Modify observations.py to maintain a small rolling buffer of the last 3 occupancy grids per environment
  • Stack them as channels before the CNN encoder (shape changes from [1,H,W] to [3,H,W] — update the encoder's first conv layer input channels accordingly)
  • Checkpoint: feed a hand-crafted sequence (obstacle moving left) through the encoder and sanity-check the output changes meaningfully between a moving vs static input — this can be tested standalone before retraining anything.
Days 5-7

Risk-prediction head + retraining

Why retrain from scratch rather than fine-tune the phase 01 checkpoint: the observation shape changed (single frame → stacked frames), so the old policy's input layer is incompatible. This is expected, not a setback — phase 01's value was proving the pipeline works, not the specific weights.
  • Add a small auxiliary output head to the policy/critic network predicting collision probability within N steps, in your skrl model definition
  • Add the binary cross-entropy auxiliary loss alongside the PPO loss, and the risk-shaped reward term in rewards.py
  • Launch training with dynamic obstacles active
  • Checkpoint: TensorBoard shows the auxiliary risk loss decreasing (the head is learning to predict something) alongside the main reward trending up.
Days 8-9

Validation

  • Run play.py with dynamic obstacles active — does the robot visibly deflect around moving things rather than freezing or colliding?
  • Extend eval_metrics.py with a dynamic-obstacle collision-rate column, compare against the phase 01 static-only baseline
  • Checkpoint, end of phase 02: collision rate with dynamic obstacles is meaningfully better than a policy with no risk head (worth training that ablation briefly, even just for a day, to have a number to compare against).

Phase 03 · ~1-2 weeks

Multi-objective optimization

So far there's been one reward, implicitly weighted by whatever numbers you picked. This phase makes that weighting explicit and deliberately varied — the deliverable is a small Pareto front showing the tradeoff between speed, energy, and smoothness, not just one more trained policy.

Why build it this way

Why decompose the reward instead of just tuning weights once? A single fixed weighting gives you one point in a tradeoff space — you can't say anything about energy versus speed from one number. Logging the four terms (progress, energy, jerk, time) separately, and training a few different weightings, gives you several points you can actually plot against each other. That comparison is the multi-objective result your supervisor asked for.
Why fixed-weight sweeps instead of a single weight-conditioned policy? A weight-conditioned policy (where the reward weights are part of the observation, randomized during training) is the more elegant solution — one policy, queryable at any tradeoff point. It's also roughly double the complexity: you need a weight-sampling curriculum and a conditioning architecture, and it's much harder to debug if something's off, since you can't easily tell whether a bad result is a training bug or just an under-sampled weight region. Fixed-weight sweeps are slower in wall-clock GPU time but far easier to reason about — get this working first, treat the conditioned version as a stretch goal only if time remains.
Why is energy "free" to compute? Your frozen locomotion policy already computes 12 joint torques every step to walk — that's its whole job. Energy is just Σ|torque × joint velocity| summed over the 12 joints and over time. You're not computing anything new, just reading and logging a number that already exists inside the simulation step.

What's added in Phase 03

The policy architecture itself (CNN + MLP, action output) is identical to phase 02 — only the reward function and the training config change, run multiple times with different weight files. No new network code is strictly required for the fixed-weight version.

Day-by-day — Phase 03

Days 1-2

Decompose and instrument the reward

  • Split the existing reward in rewards.py into four named functions: progress, energy, jerk, time — each returning its own scalar
  • Log each one separately to TensorBoard (in addition to the weighted sum) — this is the single most important habit this phase, since it's what makes the Friday-morning comparison possible
  • Move the weight values out of the reward functions and into the agent config file, so changing weights doesn't require touching code
  • Checkpoint: re-run a short training session with your phase-02 weights and confirm the four logged curves sum to (approximately) the original combined reward curve — this validates the decomposition didn't change behavior.
Days 3-7

Train the weight sweep

Why this is mostly waiting, not coding: each weight configuration is a full training run. If one run takes a day or so to reach a reasonable policy, three to five configs is the bulk of this phase's calendar time — plan other work (writing up phase 1-2 results, preparing the eval script) to run in parallel rather than sitting and watching TensorBoard.
  • Define 3-5 weight configurations: e.g. "speed-prioritized," "energy-prioritized," "balanced," "smoothness-prioritized"
  • Launch each as a separate training run, saving checkpoints with names that track which weight set produced them
  • Checkpoint per run: reward curves for all four logged terms look sane (none flatlined at zero, none exploding) within the first hour — catch a bad config early rather than discovering it after a full overnight run.
Days 8-10

Evaluate and plot

  • Run eval_metrics.py on each of the 3-5 trained checkpoints, same N episodes, same scene, for a fair comparison
  • Plot the resulting metrics against each other (e.g. energy vs. time-to-goal) — each trained policy is one point
  • Checkpoint, end of phase 03: a plot showing a visible tradeoff — e.g. the energy-prioritized policy is slower but uses less power than the speed-prioritized one. That tradeoff existing and being demonstrable is the actual deliverable.

Phase 04 · ~2+ weeks, research frontier

Multi-robot communication

Multiple Go2s share the scene, each running the full phase 1-3 stack independently, with one addition: they exchange small learned messages about what they see and where they're headed. This is explicitly the thesis-frontier part of the project — staged so even a partial result is a real, demonstrable finding.

Why build it this way

Why start with a no-communication baseline (parameter sharing) before adding messaging? "Multiple robots in one scene, each running an independent copy of the same policy" is almost free to get working — it's mechanically just instantiating your phase 1-3 pipeline more than once. It's also a legitimate scientific baseline: if you later show robots with communication do better than this, that comparison is your actual contribution. Skipping straight to the communication channel means you can't isolate whether it helped.
Why a graph neural network for the communication channel? Each robot is naturally a node, and "robots within range can hear each other" is naturally a graph edge. A GNN (or even simple attention over neighbor messages) is built for exactly this: aggregating a variable number of neighbor inputs into one fixed-size vector, which is then just one more thing concatenated into the existing nav policy's observation. This is the architecture used across current decentralized multi-robot navigation research — robots compute a compact message about their local state, send it to neighbors, and an aggregation layer folds incoming messages into each robot's own decision.
Why skrl might not be enough here, on its own? skrl's PPO/SAC/TD3 trainers are built around the assumption of one agent learning in one environment. Multiple robots that need to exchange information mid-episode is a genuinely different training loop — this is where you may need either Isaac Lab's newer multi-agent extensions or a custom wrapper, which is part of why this phase is scoped as "even reaching step 1 is a meaningful result," not "fully match the literature."

What's added in Phase 04

inter-robot communication

Each "nav policy" box at the bottom is the exact same architecture from phases 01-03 — same CNN encoder, same reward terms, same frozen locomotion underneath. The only new input is the aggregated neighbor vector. This is why a staged build (parameter-sharing first) is realistic: stages 1-3 of this phase don't touch that bottom row at all.

Staged plan — Phase 04

This phase is "day-by-day" in a looser sense than 01-03, since it's genuinely open-ended research work. Treat these as stages, each gated on the previous one actually working.

Stage A — ~3-4 days

Parameter-sharing baseline, no communication

  • Instantiate 2-3 Go2s in the same scene, each running an independent copy of your phase 01-03 nav policy (same weights, separate observations)
  • Verify they don't interfere with each other's physics/collision in ways that break locomotion
  • Checkpoint: multiple robots each independently navigate toward their own goals, ignoring each other except for incidental collision avoidance via the existing obstacle-occupancy term (they'd appear in each other's occupancy grids as just another obstacle, with zero extra code).
Stage B — ~1 week

Add the message-passing wrapper

  • Decide on the multi-agent training infrastructure: check Isaac Lab's current multi-agent support, or build a thin custom wrapper around your existing single-agent env if needed
  • Add a small encoder that compresses each robot's cost-map + state into a fixed-size message vector
  • Implement neighbor-range logic (which robots can "hear" which) and a basic aggregation (even a simple average of neighbor messages before attempting full graph attention)
  • Checkpoint: each robot's observation now includes a non-trivial aggregated neighbor vector that changes when neighbors move — verify this mechanically before training anything new.
Stage C — ~1+ week, stretch

Train and compare against the baseline

  • Train the communication-enabled policy
  • Compare against Stage A's baseline on the same metrics from phase 03 (time, energy, jerk) plus any multi-robot-specific metric (e.g. did they avoid redundant coverage, or coordinate around a shared obstacle)
  • This is genuinely a "thesis chapter," not a guaranteed week's output — reaching Stage A alone, with Stage B as documented future work, is still a legitimate result.

Metrics reference (used from Phase 03 onward)

Time-to-goal
steps × dt

Episode length until goal-reached termination fires.

Energy
Σ_t Σ_i |τ_i,t · ω_i,t|

Summed over all 12 joints and all timesteps. Read from the frozen locomotion policy's internal state.

Jerk
mean_t ‖cmd_t − cmd_{t−1}‖

Mean step-to-step change in the nav policy's velocity command.
